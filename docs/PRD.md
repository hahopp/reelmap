## 1. 한 줄 정의

**인스타에서 본 그 장소가 어디인지 알려주고, 주제별 지도로 모아 관리·공유하는 서비스.**
(포지셔닝은 "장소 일반(주제별)"이되, 첫 버티컬·시딩은 캠핑.)
첫 버티컬은 **캠핑 — 공개 등록 캠핑장만** (글램핑·오토캠핑·감성캠핑 위주). *노지/차박은 의도적으로 제외 (10장).*

---

## 2. 풀려는 문제 (Pain)

1. 인스타 인플루언서가 좋은 캠핑장을 올리지만 **어느 캠핑장인지 안 알려줌**.
2. 알려면 **팔로우 → 댓글 → DM** 의 번거로운 과정을 거쳐야 함.
3. 어렵게 알아내도 **정리할 곳이 없어** 나중에 다시 못 찾음. ← 숨은 핵심 페인

> 트레이드오프 메모: 등록 캠핑장은 예약사이트에서 어느 정도 찾을 수 있어 1건당 페인 강도는 노지보다 약함.
> 대신 **볼륨(수천 개) + 매칭 난이도(릴에 이름 없음) + 정리/여행계획 가치 + 안전 + 수익화(예약 제휴)**로 보상한다.

## 3. 핵심 설계 원칙 (이거 3개가 기획의 뼈대)

### 원칙 1 — 기여는 "보상"이 아니라 "자기 저장의 부산물" (Waze 모델)
사용자가 **남 주려고** 위치를 입력하는 게 아니라, **내가 나중에 가려고 내 지도에 저장**하는 행위 그 자체가 커뮤니티 기여가 되도록 설계한다.
→ "내 지도에 저장한다" 동선과 "커뮤니티 후보로 등록한다" 동선이 **하나**여야 한다.
→ 리워드는 그 위에 얹는 **보너스**이지, 기여를 일으키는 주 동력이 아니다.

### 원칙 2 — 인스타 스크래핑 없이 링크 매칭
인스타 URL에서 **게시물 ID만 정규화**해서 key로 사용한다 (`/p/{id}`, `/reel/{id}`).
→ 메타 API/크롤링 불필요. "이 게시물 ID → 등록된 캠핑장 후보들"을 우리 DB에서 조회만 한다.
→ 우리가 쌓는 핵심 데이터(게시물↔장소)는 **사실 정보**라 저작권 대상이 아님.

### 원칙 3 — MVP에서는 인스타 미디어를 재노출하지 않는다
썸네일/이미지/캡션 미리보기를 띄우는 순간 저작권·ToS 리스크가 커진다.
→ MVP는 **"링크 + 크라우드소싱된 장소"만** 다룬다. 우리는 콘텐츠 복제자가 아니라 그 위의 **주석(annotation) 레이어**다.

## 4. 핵심 루프 (이게 도는지를 MVP로 검증한다)

```
인스타 캠핑 콘텐츠 발견
   │
   ▼
링크 붙여넣기 ──► 게시물 ID 정규화 ──► 후보 캠핑장 리스트 조회 (신뢰도 라벨 표시)
   │
   ├─ 후보 있음 ──► 선택 ──► 내 지도에 핀 저장  ─┐
   │                                              │ (이 "저장" 자체가 그 후보에 대한 +1 투표)
   └─ 후보 없음 ──► 새 장소 추가(지도 지정/검색 + 유형/메모) ──► 저장 ──┘
                     (= 커뮤니티 후보로도 등록됨)
```

**검증할 핵심 가설**
- (가설 A) 사람들은 자기 지도에 저장하는 김에 위치를 기여한다.
- (가설 B) 나중에 같은 링크를 넣는 사람은 즉시 위치를 얻어 가치를 느낀다.

## 5. MVP 기능 범위 (스코프 규율)

### ✅ Must — MVP에 반드시
- 인스타 링크 입력 → 게시물 ID 정규화 → **후보 캠핑장 리스트 조회** (각 후보에 **신뢰도 라벨** 표시, 8장)
- 후보 **선택 → 내 지도에 핀 저장**
- 후보 없을 때 **새 장소 추가**: 지도에서 위치 지정 + 장소 검색 + 메타(유형/메모) → 저장 & 커뮤니티 후보 등록
- **내 지도 여러 개 관리** (생성/이름/삭제), 지도 뷰 + 핀 리스트 뷰
- 지도 **공개 범위**: 비공개 / 링크공유(공개) — *친구공유는 v1.1 (10장)*
- **시드맵**(운영자가 미리 만든 공개 지도) 노출 + "내 지도로 담기"
- **신고** 버튼(틀린 정보 제보)
- 소셜 로그인(카카오/구글)

### 🔜 Should — 바로 다음 단계 (v1.1)
- **네이티브 앱 + OS 공유시트 연동** (12장 — 복붙 제거, 사실상 필수)
- **친구 공유**(친구에게만 공개), 친구 그래프
- 등급/리워드(핀·맵 개수 확장, 광고 제거)
- 기여자 정확도/신뢰 점수, 공개 기여자 프로필
- 유형별 검색·필터, 지역 검색
- **예약처 링크/제휴**(수익화)

### ❌ Won't — MVP 제외 (의도적으로 안 함)
- 노지/차박 좌표 (민감성 회피 — 공개 등록 캠핑장만)
- 인스타 이미지/캡션 미리보기, 자동 크롤링
- 네이티브 앱 (MVP는 반응형 웹 — 단, PWA 공유 브리지는 검토, 12장)
- 결제/광고/포인트 정산

## 6. 화면 흐름 (5개 핵심 화면)

1. **홈** — 링크 붙여넣기 바 + 내 지도 목록 + 추천 시드맵
2. **링크 조회 결과** — 후보 리스트(캠핑장명·유형·**신뢰도 라벨**) + "원하는 게 없어요 → 새로 추가"
3. **장소 추가** — 지도 picker + 장소 검색 + 유형 선택(카테고리별·룩업 기반; 캠핑: 글램핑/오토캠핑/카라반/일반/기타) + 메모 + 저장할 지도 선택
4. **내 지도 상세** — 지도 + 핀 리스트, 공개범위 설정, 공유 링크 복사
5. **공개/시드 지도 보기** — 읽기 전용 + "내 지도로 담기"

## 7. 데이터 모델 (핵심 엔티티)

> 크라우드소싱의 심장은 역할이 분리된 세 테이블이다:
> - `submission` = **후보 정본**(콘텐츠×장소 1행 — 신고·집계·시드 라벨의 단위)
> - `selection` = **투표**(누가 그 후보를 지지하나 — 유저당 1개)
> - `map_pin` = **담기**(어느 지도에 올렸나 — 지도별 N개 가능)
> **신뢰도 = 한 `submission`을 지지한 고유 유저 수(distinct selection.user_id).**

| 엔티티 | 핵심 필드 | 비고 |
|---|---|---|
| **user** | id, 소셜계정, 닉네임, created_at | |
| **content** | id(정규화된 post_id), source_url, platform(instagram), created_at | 인스타 게시물 1개 = 1 row |
| **place** | id, name, lat, lng, **category**(camping…), **type_key**(카테고리별·룩업 참조), **external_provider?**(kakao/naver), **external_place_id?**, address?, road_address?, reservation_url?, created_by | 유형 하드코딩 X(7-1장) · (provider, external_place_id)로 중복 방지 · 좌표 WGS84 저장으로 지도 제공자 독립(7-2장) |
| **place_type** | category, key, label, sort | 유형 룩업(데이터). 확장 = 행 추가 |
| **submission** | id, content_id, place_id, submitted_by, **source**(user/seed), **status**(active/flagged/hidden), created_at · **UNIQUE(content_id, place_id)** | 콘텐츠×장소 **후보 정본 1행**. 신고·집계·시드 라벨의 단위. `source=seed`면 🏕 공식 시드 |
| **selection** | id, user_id, **submission_id**(FK), created_at · **UNIQUE(user_id, submission_id)** | 후보 지지 **투표**. UNIQUE → **1계정 1표**. content/place는 submission에서 파생, **map_id 없음** |
| **map** | id, owner_id, title, description, visibility(private/unlisted), **share_token**, is_seed | 시드맵은 운영자 소유 + is_seed=true · 공유 URL은 추측불가 `share_token` 사용(10장) |
| **map_pin** | id, map_id, place_id, content_id?, note, added_at | 유일한 "**담기**" 단위. 같은 place를 여러 지도에 담으면 핀만 N개(투표는 1) |
| **report** | id, target_submission_id, reason, reporter_id, created_at | 틀린 정보 제보. 누적 시 `submission.status`를 flagged로(8장) |

**핵심 동작 매핑**
- 링크 조회: `SELECT sub, place, COUNT(DISTINCT sel.user_id) FROM submission sub JOIN place ... LEFT JOIN selection sel ON sel.submission_id = sub.id WHERE content_id = ? AND status='active' GROUP BY sub.id ORDER BY 지지수 DESC`
- 후보 선택해서 저장: `selection` INSERT(`ON CONFLICT(user_id, submission_id) DO NOTHING`) + `map_pin` INSERT
- 새 장소 추가: `place` UPSERT(**kakao_place_id 기준 dedup**) + `submission` UPSERT(content_id+place_id) + `selection` INSERT + `map_pin` INSERT

### 7-1. 확장 설계 — 카테고리 / 유형 (지금 잡아두는 이유)
- 유형 목록을 **하드코딩하지 않는다.** `category`(대분류) + `type_key`(소분류, `place_type` 룩업 참조) 2단 구조.
- 향후 확장(호텔·펜션·리조트·식당 등) = **`place_type`에 행 INSERT**로 끝. 스키마·코어 코드 변경 없음.
  - 예) `{camping, glamping, 글램핑}` … (나중에) `{stay, hotel, 호텔}` / `{food, korean, 한식}`
- place/submission/selection/map 핵심 테이블은 캠핑 전용이 아님 → 그대로 타 카테고리에 재사용.
- **단, 출시·시딩은 캠핑만**(콜드스타트 집중). 구조는 유연하게, 런칭은 좁게. 캠핑이 임계점 도달 후 stay→food 순으로 카테고리를 "연다".

### 7-2. 장소 중복 방지 (dedup) + 지도 제공자 독립
- 같은 캠핑장을 다른 좌표/이름으로 중복 등록하면 `place`가 쪼개져 **신뢰도 카운트가 분산**된다.
- 대응: 장소 추가는 **장소검색 결과 선택을 우선** → 결과의 `(external_provider, external_place_id)`를 저장하고 그 값으로 dedup.
  - `CREATE UNIQUE INDEX place_ext_uniq ON place (external_provider, external_place_id) WHERE external_place_id IS NOT NULL;`
  - 이미 있으면 기존 `place` 재사용(UPSERT), 없으면 신규.
- **좌표는 항상 WGS84(lat/lng)로 저장** — 카카오/네이버 지도 SDK·검색 API 모두 WGS84라 **지도 제공자를 바꿔도 좌표 데이터는 그대로**.
- **제공자 교체 용이성**: 검색·지도 호출은 `lib/places` 어댑터(`searchPlaces()`)와 `<MapView>` 컴포넌트 뒤로만 격리 → 카카오→네이버 전환 = 그 두 곳만 교체. (단 네이버 지역검색은 place id가 없어, 네이버 신규 등록분은 `name+road_address`/좌표 근접으로 dedup)
- 검색에 없어 직접 좌표를 찍은 경우 `external_place_id=null` → dedup 제외, 운영자 병합 후보로 남긴다.

## 8. 신뢰도(Trust) 모델

- **기본 신호 = 고유 사용자 선택수** (그 후보를 자기 지도에 담은 서로 다른 사람 수). 소셜로그인으로 1계정 1표.
- **숫자 대신 상태 라벨로 노출** (사람은 숫자보다 라벨을 신뢰):
  - 🆕 **미확인** — 등록자 1명. "아직 확인 안 됨, 참고용"
  - 👥 **N명 선택** — 여러 명 동의
  - ✅ **확인됨** — 임계치 이상(예: 5명). 신뢰 배지
  - 🏕 **공식 시드** — 운영자 시드 출처(`submission.source='seed'`, 지지수와 무관하게 우선 표시)
- **충돌(같은 콘텐츠, 다른 장소)**: 선택수 내림차순 정렬. 1·2위가 근접하면 "여러 의견 있음" 표시 → 사용자가 판단.
- **등록자(기여자) 표시**: 누가 처음 등록했는지 노출 → 추후 평판/리워드와 연결.
- **어뷰징 방어(MVP 최소)**: 1계정 1표 + 신고. **신고는 자동 삭제가 아니라 검토 플래그** — 서로 다른 신고자(distinct reporter)가 임계치에 도달하면 `submission.status='flagged'`(검토 큐), 높은 임계치에서만 `hidden`. (악의 신고로 정상 후보가 사라지는 사고 방지) 정교한 점수 공식·정확도 가중치는 v1.1.
- **익명 인증의 sybil 한계 (중요)**: Phase 2의 Supabase 익명 인증은 누구나 무제한 익명 ID를 만들 수 있어 **익명 단계의 신뢰도 카운트는 부풀리기 가능**하다. 따라서 **정식 신뢰도 집계는 소셜 로그인(Phase 3) 유저 기준**이며, 익명 단계 수치는 "흐름 검증용"으로만 본다(M2 게이트 정의와 일치).

## 9. 캠핑 버티컬 시딩 전략 (콜드스타트 대응)

- 시딩 대상: **글램핑 · 오토캠핑 · 감성캠핑 · 카라반** 등 공개 등록 캠핑장
- **시딩 도구 = 어드민 콘솔(`/admin`, 비밀번호 게이트)**: 지도 CRUD + 인스타 링크·장소 등록(장소검색으로 좌표 확정) + 지도에 핀 추가/제거. 운영자가 이걸로 **기획전 성격의 public 시드맵**을 직접 구축한다. (소비자 핵심 루프보다 먼저 만든다 — `DEVELOPMENT_PLAN.md` 작업 트랙)
- 유명 캠핑 인플루언서 2~3명 선정 → "○○님이 다녀온 캠핑장 지도"를 직접 제작 → 공유 자체가 광고
- **★ 콘텐츠 매핑까지 시딩한다 (가설 B의 전제):** 장소만 모으지 말고, 인플루언서의 **실제 릴/게시물 URL을 `content`로 등록**하고 해당 캠핑장으로 `submission(source='seed')`을 생성한다. → 사용자가 그 릴을 붙여넣는 순간 **즉시 후보가 떠 재사용 가치(가설 B)가 실증**된다. (장소 지도만 시딩하면 임의의 새 링크는 여전히 후보 0)
- 계절·테마 지도로 확장: "봄에 갈 감성 캠핑장", "아이랑 가기 좋은 오토캠핑장" 등
- 첫 목표: **테마 시드맵 1~3개**로 초기 유입 만들기
- **온보딩**: 빈 상태(내 지도 0개)에서도 시드맵이 즉시 가치를 주도록 홈 상단에 노출 → "담기"로 첫 액션 유도

## 10. 공개 범위 정책 & 스코프 결정

- **공개 범위**: MVP는 `비공개(나만)` / `링크공유(unlisted)` 2단계. v1.1에서 `친구에게만` 추가 → 최종 3단계.
  - `링크공유`는 "**링크를 아는 사람만**"이다(검색 노출되는 public 아님). 정수 id 추측을 막기 위해 공유 URL은 추측 불가능한 **`map.share_token`** 사용: `/m/{share_token}`.
  - **검색 인덱싱:** MVP엔 진짜 public(검색노출) 등급이 없다 → **모든 유저 지도 페이지는 `noindex`**, OG/인덱싱은 **운영자 시드맵(`is_seed`)에만** 허용. ("비공개로 알고 공유했는데 검색에 뜸" 사고 차단)
- **노지/차박 제외**: 사유지·금지구역·환경보호구역 민감성 회피. **공개 등록 캠핑장만** 다룬다. (안전·지속가능·업체 친화·수익화 경로 확보)

## 14. 지표 / 이벤트 로깅 (가설 검증용)

### 원칙
- **"이벤트는 질문에 답하려고 존재한다."** 다 찍지 말고, 검증할 질문에 매핑되는 것만.
- 답할 질문: (A) 기여가 일어나나 · (B) 재사용 가치가 생기나 · (성장) 시드→가입→활동.

### 방법
- 이벤트 이름 = `객체_동작` snake_case, 일관 유지.
- 이벤트마다 **속성(properties)** 부착해 맥락 저장(비율 계산용).
- 익명 ID → 로그인 시 실제 유저에 **alias(연결)** → 가입 전후 추적.
- 발화: UI 액션은 클라이언트, 신뢰 중요한 것(`place_created`,`selection`)은 **서버**에서.
- 도구: **PostHog**(제품 퍼널·무료티어) 추천, GA4는 유입 보조.
- MVP는 핵심 이벤트 8~9개로 시작, 과설계 금지.

### 핵심 이벤트
| 이벤트 | 언제 | 주요 속성 | 답하는 질문 |
|---|---|---|---|
| `signup_completed` | 가입 완료 | method, ref_seed_map_id? | 성장 |
| `link_looked_up` | 링크 조회 실행 | had_candidates, candidate_count, top_trust_label | 가설 B |
| `candidate_selected` | 기존 후보 선택해 저장 | content_id, place_id, trust_label, map_id | 가설 B |
| `place_created` | 새 장소 등록(=기여) | content_id, place_type, from_empty | 가설 A |
| `map_pin_saved` | 핀이 지도에 저장됨 | map_id, via(candidate/new) | 활동량 |
| `map_shared` | 공유 링크 생성/복사 | map_id, visibility | 바이럴 |
| `seed_map_viewed` | 시드맵 열람 | seed_map_id | 유입 |
| `seed_pin_added` | 시드맵에서 내 지도로 담기 | seed_map_id, place_id | 시드→활성 전환 |
| `report_submitted` | 신고 | target, reason | 데이터 품질 |

### 핵심 퍼널 & 지표
```
seed_map_viewed → signup_completed → link_looked_up → (candidate_selected | place_created) → map_shared
```
- **기여율(후보 없음)** = `place_created` / (`link_looked_up` 중 had_candidates=false) ← 가설 A 주지표
  - 보조: **후보 있어도 새로 추가** = `place_created`(from_empty=false) / (`link_looked_up` 중 had_candidates=true). 후보가 있어도 기여하는 행동이라 가설 A에 같이 본다.
- **재사용 가치율** = `candidate_selected` / (`link_looked_up` 중 had_candidates=true) ← 가설 B
- **시드 전환율** = 시드 유입 → 가입
- **주의(이중계상):** `map_pin_saved`는 `candidate_selected` + `place_created`의 저장분을 모두 포함하는 상위집합. "저장 총량"엔 `map_pin_saved`, 가설 A/B 분해엔 `candidate_selected`/`place_created`를 쓰고 **합산하지 말 것.**

## 15. 프로토타입(b단계) 추천 스택

- **로그인**: 카카오/구글 소셜 + **Supabase 익명 인증**(Phase 2부터 익명 ID로 1계정 1표 동작 → Phase 3에서 소셜로 link 승격)
- **분석**: PostHog
- **호스팅**: Vercel + Supabase
