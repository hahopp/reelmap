# ADR 0003 — 태그 모델: 2축 태깅(유형/속성) [제안 · 보류]

> **상태**: 제안됨 (**보류** — 데이터·유저 볼륨 생기면 재검토) · **날짜**: 2026-06-21 · **관련**: [0002-instagram-ingest-pipeline.md](0002-instagram-ingest-pipeline.md) · `place_tag` · [../STATUS.md](../STATUS.md) §4

## 맥락
서비스는 주제 일반("인스타 주소 → 주제별 지도", 캠핑은 첫 시드). 현재 태그는 **자유 형식**이고 통제 어휘 테이블 `place_tag`는 **빈 상태**다. 논의 주제: 음식점/캠핑장/리조트 같은 **"장소 종류(유형)"** 를 태그 체계에서 어떻게 다룰까. 유형은 **지도 핀 아이콘 구분·"~만 보기" 필터·일관성**에 필요해질 수 있다(사용자가 "인증태그/속성 부여" 형태로 요청).

## 결정 (제안 — 지금은 보류)
태그를 **2축**으로 나눈다:
- **유형(kind)**: "이게 *무엇*인지". **통제 어휘에서만 · 장소당 1개(대표) · 확정 시 필수.** `place_tag.category='type'` 로 표시.
- **속성(descriptor)**: "*어떤*지"(오션뷰·감성·키즈…). **자유 생성 · 여러 개.**

저장은 **`place.tags text[]` 하나 유지**(유형도 거기 들어가되 `place_tag`로 역할 구분). **별도 category 컬럼은 만들지 않음**(ADR 0002의 category/type_key nullable·중립화와 일관).

**사용자도 유형 추가 가능 — 단 "제안 → 승격"**(folksonomy → controlled vocabulary):
- 누구나 새 유형 *제안* → `pending` 상태로 장소엔 붙되 **공식 facet·아이콘엔 미노출**.
- 운영자가 **승격(canonical) / 병합(맛집→음식점) / 반려**. "공식 어휘에 있음 = 인증됨"(별도 `verified` 배지 안 만듦).
- **단계적**: 운영자 단독기엔 운영자가 직접 canonical 추가. 소비자·AI 제안이 생기면 **검수 큐** 활성화.

**유형 인정 4테스트(모두 YES여야 유형)**: ① 따로 필터/아이콘 줄 만한가(구분 행동) ② 정체인가 특징인가(본질) ③ 다른 유형과 같은 층위인가(음식점 O / 파스타집 X) ④ 실제 쌓일 만한가. → 하나라도 NO면 **속성**.
**유형 아닌 것**: 하위유형(글램핑·한식) · 분위기/대상/지역(감성·키즈·수도권) · 맵 주제("감성카페투어"는 지도지 유형 아님).

## 근거
- 유형은 **저카디널리티·고가치**(아이콘·facet 일관성) → 통제해야. 속성은 롱테일 → 자유로워도 무해. 둘을 같은 규칙으로 다루면 제일 중요한 필터가 흩어져 깨짐.
- 한 장소가 복수 종류일 수 있어(카페 겸 서점) **배타적 컬럼보다 태그-역할이 유연**.
- **"사용은 열고 공식화만 통제"** 가 "유저도 추가 + 일관성 유지" 둘 다 얻는 유일한 방법(자유 편집 = 통제 깨짐).

## 영향 / 보류 사유
- **지금은 미구현(보류)**: 현 단계(시드·데이터 적음·공개 유저 없음)에선 태그 흩어짐이 아직 실제 문제가 아님 → **과투자**. 기존 자유 태그로 진행.
- **도입 시 필요한 것**: `place_tag.status`(canonical|pending) 컬럼 + 유형 v1 시드(예: 음식점·카페·바·숙소·캠핑·전시·자연명소·쇼핑·기타) + 확정/편집 UI에 **유형 단일 select** + (소비자기) **검수 큐** + 지도 핀 아이콘. 유형 변경은 재태깅 비용 크니 **넓고 안정적으로, 세분화는 속성으로**.
- **재검토 트리거**: 데이터·유저 볼륨이 늘어 태그검색/핀 아이콘 일관성이 실제로 아쉬워질 때.
