> 사용자 요청으로 **운영자 시드 도구(어드민)부터** 만들고 그 데이터를 공개 지도로 보여준 뒤, 소비자 핵심 루프로 간다. 아래가 실제 진행 순서이며, 기존 Phase 0~5는 참고용 백로그로 둔다.

| 순서 | 내용 | 매핑 |
|---|---|---|
| ✅ T0 | 스캐폴드(Next16/React19/Tailwind4) + 기반 정리 | Phase 0 |
| **T1** | Supabase 연결 + 스키마 적용(7장 개정판) | Phase 1 앞당김 |
| **T2** | **어드민 콘솔 `/admin`**(비밀번호 게이트) — 지도 CRUD · 인스타링크+장소 등록(카카오 검색→좌표·dedup) · 지도 핀 추가/제거. 저장물=`submission(source='seed')`+`map_pin`, `is_seed` 지도 | Phase 4 시드 + 신규 |
| **T3** | **기획전 public 시드맵 콘텐츠 채우기**(운영자가 T2로 입력) | Phase 5 시드 채우기 |
| **T4** | **Public 지도 뷰**(비로그인 열람, 지도에 핀 렌더, 유저맵 noindex·시드맵 인덱싱) | Phase 4 공개뷰 |
| 이후 | 소비자 핵심 루프(링크 조회→선택/추가→내 지도), 인증, 신고 등 | Phase 2·3 본류 |

**결정(2026-06-15)**: 지도/검색은 **카카오로 시작**(검색 결과 多·place id로 dedup 유리) + `lib/places` 어댑터로 격리 → 추후 네이버 교체는 어댑터+지도 컴포넌트 교체로 한정(좌표 WGS84 저장이라 데이터 마이그레이션 불필요). 어드민 보호 = **env 비밀번호 게이트**(단일 운영자 MVP).

## 개요 — 단계 설계 원칙

1. **가장 위험한 걸 먼저 검증한다.** 제품 최대 리스크 = "사람들이 기여/재사용하나(가설 A·B)" → **핵심 루프(Phase 2)를 최우선**으로 동작시킨다.
2. **인프라 먼저.** 배포·DB·키를 Phase 0에 깔아 "로컬에서만 됨" 리스크 제거.
3. **데이터 → 백엔드 → UI 순.** 스키마/쿼리를 UI보다 먼저 확정.
4. **각 단계는 검증 가능한 산출물(DoD)로 끝난다.**

### 마일스톤
| 마일스톤 | 끝나는 시점 | 의미 |
|---|---|---|
| **M1** | Phase 1 | 인프라 + 데이터 코어 준비 완료 |
| **M2 ⭐** | Phase 2 | **핵심 루프 동작 — 가장 중요한 검증 게이트** (=앞서 말한 (1)핵심루프) |
| **M3** | Phase 4 | 실유저·공유·시드 갖춘 기능 완성 |
| **M4** | Phase 5 | 베타 출시 & 지표 측정 시작 |

> 효율 표기 S/M/L은 상대 크기(절대 기간 아님). 1인 개발 기준 대체로 순차 진행.
> 병렬 가능: **시드맵 콘텐츠 리스트업**은 개발과 무관하게 아무 때나 진행 가능.

## Phase 0 — 프로젝트 세팅 (Foundation) · [S]

**목표**: 배포·DB·키 인프라를 먼저 깔아 인프라 리스크 제거.

- [ ] GitHub 레포 생성, Next.js(App Router + TypeScript) 스캐폴드
- [ ] Vercel 연결 → `main` push 시 자동 배포 확인 (빈 페이지라도 URL 뜨게)
- [ ] Supabase 프로젝트 생성, 환경변수(.env.local + Vercel env) 세팅
- [ ] 카카오 개발자 앱 등록 → JavaScript 키 발급 + 서비스 도메인 등록
- [ ] PostHog 프로젝트 생성, API 키 확보 (연동은 Phase 5)
- [ ] 기본 레이아웃/라우팅 뼈대 + 디자인 토큰(색·폰트·간격) 최소 정의

**DoD**: 배포된 URL에서 기본 페이지가 뜨고, Supabase 연결 OK, env가 로컬/배포 양쪽에서 동작.

## Phase 1 — 데이터 모델 & 코어 백엔드 · [M]

**목표**: 스키마와 핵심 쿼리를 UI보다 먼저 확정.

- [ ] 테이블 생성: `user, content, place, place_type, submission, selection, map, map_pin, report` (스키마는 PRD 7장 개정판 기준)
- [ ] `place_type` 캠핑 유형 시드: 글램핑/오토캠핑/카라반/일반/기타 (category=camping)
- [ ] 제약: `submission UNIQUE(content_id, place_id)`(후보 정본) + `selection UNIQUE(user_id, submission_id)` → **1계정 1표**
- [ ] `place` dedup: `kakao_place_id` 부분 유니크 인덱스 + 추가 시 UPSERT (PRD 7-2장)
- [ ] **인스타 URL → post_id 정규화** 함수 (`/p/`, `/reel/`, `/tv/`, `/{username}/reel/{id}`, `?igsh=`·`?utm_` 트래킹 파라미터·트레일링 슬래시 제거, 대소문자) + 테스트 케이스
- [ ] 후보 조회 쿼리: `content_id` → submission+place + `COUNT(DISTINCT selection.user_id)` 내림차순, `status='active'`만
- [ ] 신뢰도 라벨 계산 로직 (미확인 / N명 선택 / 확인됨(≥임계치) / 🏕공식시드=`source='seed'`)
- [ ] 기본 RLS 정책 초안: 공개 집계(place/submission/selection 읽기 공개), **map/map_pin은 visibility 분기**(private=본인만, unlisted=공개), 쓰기는 본인만

**DoD**: 간단 스크립트/SQL로 "링크 넣으면 후보 + 신뢰도가 나온다"가 데이터 레벨에서 검증됨.

## Phase 2 — 핵심 루프 UI (MVP의 심장) · [L] ⭐

**목표**: 링크 조회 → 선택/추가 → 내 지도 저장을 end-to-end로. **가설 A/B 검증 지점.**

- [ ] 홈: 링크 입력 바 + 인스타 URL 형식 검증
- [ ] 조회 결과 화면: 후보 리스트(이름·유형·신뢰도 라벨) + 후보 없음 빈 상태
- [ ] 카카오맵 연동: 지도 표시, 마커, 키워드 장소검색(→좌표), 클릭으로 좌표 지정
- [ ] 새 장소 추가 플로우: 지도 picker + 검색 + 유형 선택(룩업 기반) + 메모 → 저장
- [ ] 후보 선택 → 내 지도에 핀 저장 (`selection` + `map_pin` 생성)
- [ ] 내 지도 상세(임시 버전): 지도 + 핀 리스트
- [ ] **Supabase 익명 인증**으로 동작 — 디바이스별 익명 ID 부여(distinct user/1계정 1표 로직을 익명 단계에서도 실제 동작). 소셜 로그인은 Phase 3

**DoD**: **"링크 조회 → 후보 선택/새 장소 추가 → 내 지도에 핀이 남는다"** 흐름이 배포 URL에서 end-to-end 동작. ← **M2 게이트(= 흐름 검증). 가설 A/B 정량 검증은 실유저가 쌓이는 Phase 3 이후로 본다.**

## Phase 3 — 인증 & 내 지도 관리 · [M]

**목표**: 실제 유저 + 1계정 1표 + 다중 지도.

- [ ] 소셜 로그인(카카오/구글) via Supabase Auth
- [ ] **익명 → 영구 계정 link**(Supabase identity linking) — Phase 2의 익명 데이터가 그대로 승격 (alias 숙제 자동 해결)
- [ ] 내 지도 CRUD: 생성/이름변경/삭제, 여러 개 관리
- [ ] 저장 시 "어느 지도에 담을지" 선택 UI
- [ ] `selection` UNIQUE로 1계정 1표 실제 적용
- [ ] RLS: 본인 데이터만 쓰기/수정

**DoD**: 로그인 유저가 자기 지도 여러 개를 만들고 핀을 담을 수 있고, 같은 후보 중복 투표가 막힌다.

## Phase 4 — 공유 · 시드맵 · 신고 · [M]

**목표**: 바이럴 + 콜드스타트 시드 + 데이터 품질.

- [ ] 공개 범위(비공개/링크공유) 설정 + **`share_token` 기반 공유 링크**(`/m/{share_token}`) 생성/복사
- [ ] 공개 지도 읽기 전용 뷰 (비로그인도 열람 가능) — 유저 지도는 `noindex` / 시드맵만 OG·인덱싱
- [ ] 시드맵: `is_seed` 지도 홈 노출 + "내 지도로 담기"
- [ ] 신고 버튼 + `report` 저장 + distinct reporter 임계치 → `submission.status='flagged'`(검토 큐, 높은 임계치에서만 hidden)

**DoD**: 공유 링크로 남이 내 지도를 보고, 시드맵을 자기 지도로 담을 수 있고, 신고가 쌓인다. ← **M3 게이트**

---

## Phase 5 — 계측 · 다듬기 · 베타 출시 · [M]

**목표**: 가설 측정 + 출시.

- [ ] PostHog 9개 이벤트 연결 + 속성 부착, 익명→유저 alias
- [ ] 퍼널/지표 대시보드 (기여율 · 재사용가치율 · 시드전환율)
- [ ] 빈 상태·로딩·에러 처리, 모바일 반응형 점검
- [ ] **시드맵 콘텐츠 1차 채우기** (인플루언서 1~2명, 캠핑장 N개) ← 개발과 병렬 가능
- [ ] 기본 SEO/OG 태그(시드맵만 인덱싱·유저 지도 noindex) + 공유 링크 미리보기 + 도메인 연결
- [ ] 베타 배포 → 지인 테스트 → 피드백 루프

**DoD**: 핵심 이벤트가 PostHog에 들어오고, 시드맵이 채워진 상태로 베타 URL을 공유 가능. ← **M4 게이트**

## 진행 순서 요약

```
Phase 0 (세팅)
   └─► Phase 1 (데이터/백엔드)  ───────────── M1
          └─► Phase 2 (핵심 루프 UI) ⭐ ────── M2  ← 여기서 가설 A/B 1차 검증
                 └─► Phase 3 (인증/내 지도)
                        └─► Phase 4 (공유/시드/신고) ── M3
                               └─► Phase 5 (계측/출시) ── M4
( 시드맵 콘텐츠 리스트업: 위 전 과정과 병렬로 언제든 )
```

## 참고 — 결정 사항 반영
- 앞선 "프로토타입 범위" 질문은 이 단계 계획으로 해소됨: **Phase 2 끝 = (1)핵심 루프 검증 마일스톤(M2)**. 이후 점진 확장.
- 카카오맵은 Phase 0에서 키 발급 → Phase 2에서 실연동(기본 가정). 더 빨리 흐름만 보고 싶으면 Phase 2를 더미 지도로 시작 후 교체 가능.
